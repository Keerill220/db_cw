РОЗДІЛ 3. РОЗРОБКА ПРОГРАМНОГО КОДУ

3.1. Обґрунтування обраної архітектури

3.1.1. Тип архітектури
Під час розробки інформаційної системи «SoundSpace» було реалізовано модульну багаторівневу архітектуру, орієнтовану на чітке розмежування функціональних відповідальностей та зручність масштабування. Архітектурна модель базується на розділенні компонентів на рівні контролерів (Controllers) та логіки обслуговування, тобто сервісів (Services). На рівні контролерів обробляються HTTP-запити до вбудованого REST API, виконується авторизація користувача на основі створеного JWT-токена та виклик відповідної функції в сервісі.
Основна логіка функціонування системи – перевірка обмежень, застосування правил бронювання та обробка бізнес-сценаріїв – реалізована на рівні сервісів. Якщо всі необхідні перевірки валідації (наприклад, доступність обраної кімнати на запитаний час) успішно пройдені, сервіси здійснюють підключення до бази даних через патерн Repository для виконання відповідних запитів, що включають: створення, читання, оновлення чи видалення записів. Обрана архітектура дозволяє легко відокремити відповідальність за взаємодію в мережі, виконання бізнес-правил та доступ до сховища. Такий підхід забезпечує зрозумілу структуру серверного коду, спрощує тестування, відлагодження та подальший розвиток функціоналу.

3.1.2. Мова програмування та система баз даних
Мовою програмування для розробки серверної частини системи було обрано C# (C Sharp) — сучасну, кросплатформну мову, яка забезпечує високу продуктивність, надійність та безпеку. Основним фреймворком для побудови API виступає ASP.NET Core 8, що підтримує асинхронну обробку запитів, має потужні механізми Dependency Injection (впровадження залежностей) та інтегровану генерацію документації через Swagger.
Для взаємодії з базою даних використовується Entity Framework Core 8 – передова ORM-бібліотека, що дозволяє виконувати зручне об'єктно-реляційне відображення, будувати гнучкі LINQ-запити, керувати транзакціями та узгодженими сутностями. Керування структурою бази реалізовано за допомогою міграцій EF Core: це полегшує її ітеративний розвиток та інтеграцію SQL-скриптів для тригерів і процедур (система зберігає скрипти у директорії SQL всередині Data-шару).
В якості сховища обрано PostgreSQL – продуктивну об'єктно-реляційну систему баз даних, що забезпечує транзакційність, високу надійність збережених процедур та тригерних функцій для підтримки бізнес-правил на нижньому рівні.
Клієнтська частина системи реалізована з використанням бібліотеки React 18, мови TypeScript та збирача Vite для побудови динамічного односторінкового додатку (SPA). Для стилізації інтерфейсів застосовано фреймворк Tailwind CSS, що дозволяє швидко та зручно генерувати адаптивний дизайн. Ці технології дозволили створити інтерактивну та сучасну систему.

3.1.3. Протоколи обміну
Обмін даними між клієнтським додатком на React та бекенд-сервером на ASP.NET Core відбувається заRESTful-архітектурою через HTTP. Стандартом для форматування переданих даних обрано JSON (JavaScript Object Notation), що сприяє зручності інтеграції та високій пропускній здатності для вебдодатків. Всі запити побудовані навколо асинхронної обробки (CancellationToken, Task), що унеможливлює блокування потоків виконання під час читання великих звітів чи списків студій.

3.2. Структурна модель інформаційної системи

Клієнтський інтерфейс (React + TypeScript)
- Забезпечує інтерактивну відображення даних та зручну взаємодію клієнта зі сторінками системи.
- Відповідає за представлення даних: динамічний перелік репетиційних баз, відображення сітки годин кімнат (календар), панелі для входу/реєстрації, а також персональні профілі з історіями бронювань.
- Реалізує маршрутизацію (за допомогою React Router 7) для розмежування прав: адміністративні панелі (AdminPage) недоступні для звичайних клієнтів.
- Усі мережеві запити (fetch або axios) перехоплюються обгортками (наприклад, у src/api), які додають JWT-токен у заголовок Authorization та обробляють відповіді сервера.

Серверна частина (ASP.NET Core Web API)
- Обробляє отримані від браузера REST-запити на різні ендпоінти.
- Включає контролери (StudiosController, BookingsController, AuthController) з розмежуванням доступу: загальні маршрути є публічними, керування бронюваннями вимагає авторизації, тоді як ReportsController чи AdminController доступні лише для адміністраторів.
- Бізнес-логіка (наприклад, перевірка збігу розкладу, розрахунки ціни обладнання, обчислення фінансових звітів) зосереджена у сервісах (BookingService, ReportService та ін.).
- Механізм авторизації та видачі ролей спирається на сервіс JwtTokenService.

База даних (PostgreSQL)
- Виступає основним сховищем для об'єктів: клієнтів, адміністраторів, студій, кімнат, обладнання, бронювань.
- Забезпечує інтегритет даних (перевірка зовнішніх ключів, унікальність електронних скриньок).
- Працює у зв'язці із внутрішніми тригерами для критичних бізнес-правил (fn_booking_no_overlap – для запобігання подвійних резервувань кімнати, fn_booking_calc_total_price – для автоматичного підсумовування загальної вартості сесії).

Взаємодія компонентів мережі:
- Інтерфейс React ініціює запит на бронювання із параметрами (кімната, дата, слот, бажане обладнання).
- Контролер ASP.NET перехоплює запит, валідує токен та передає DTO у відповідний сервіс BookingService.
- Сервіс створює об'єкти та викликає Repository, який обробляє дані через EF Core та зберігає у PostgreSQL.
- Тригери PostgreSQL під час фіксації запису додатково переконуються у відсутності накладок у часі та ініціалізують розрахунок повної вартості оренди.
- СУБД повертає готову сутність з новим ідентифікатором, а бекенд надсилає успішну JSON-відповідь назад до фронтенду.

3.4. Опис реалізації функцій системи

3.4.1. Створення та перегляд бронювань
Коли зареєстрований або авторизований клієнт переходить на сторінку кімнати, він може ініціювати процес оренди приміщення:
- обирається дата і конкретний часовий слот, який відображається як "вільний";
- за потреби додається доступне у студії музичне обладнання;
- система динамічно прораховує вартість запиту;
- після відправлення запиту база даних (за допомогою тригера no_overlap) фінально підтверджує можливість замовлення, щоб попередити конфлікти між різними клієнтами;
- в разі успіху сесія отримує статус "Pending" або "Confirmed".
Цей потік впроваджено на фронтенді через BookingPage, де зберігається стан форми, валідуються дати і взаємодіють з локальними API-обгортками.

3.4.2. Управління даними клієнтів
Процедури реєстрації та входу реалізовані через інтерфейси LoginPage та RegisterPage. Під час заповнення даних:
- здійснюється перевірка валідності полів (допустимої довжини, коректного формату пошти);
- за допомогою bcrypt (BcryptPasswordHasher) серверна частина хешує пароль перед збереженням;
- в разі успішного входу користувач отримує JWT, що зберігається в пам'яті фронтенду або контексті (AuthContext) і згодом застосовується для всіх приватних дій.
В особистому кабінеті (ProfilePage) клієнт отримує доступ до своєї історії створених бронювань із можливістю їх детального огляду або скасування, якщо це передбачено правилами скасування.

3.4.3. Експорт даних для адміністраторів
Для керування музичними базами з боку власників студій реалізовано комплексну адміністративну панель (AdminPage). У цьому розділі адміністратор:
- відслідковує активні або минулі бронювання для своєї локації;
- використовує сторінку звітів (підтримується ендпоінтами ReportsController), яка дозволяє переглядати статистику завантаженості кімнат, загальний дохід від оренд (через спеціалізовану збережену процедуру proc_occupancy_report) та виділяти найбільш прибуткові приміщення;
- може керувати каталогом студій: змінювати інформацію, додавати нові кімнати чи керувати балансами інвентарю.

3.4.4. Редагування даних
Редагування різних сутностей відбувається за допомогою захищених форм керування:
- редагування кімнати (назва, опис, площа, ціна);
- додавання та модифікація обладнання з ціноутворенням на погодинній основі;
- оновлення власного паролю або контактів;
Усі такі зміни обов'язково проходять подвійну валідацію. Спочатку на клієнті – для економії часу та попередження пустої форми, і вдруге на сервері – для підтвердження бізнес-правил та перевірки наявності таких сутностей у БД.

3.5. Опис реалізації елементів інтерфейсу
Інтерфейс «SoundSpace» розроблений на базі компонентів React 18, які організовано як набір повторно використовуваних модулів.
Основні елементи:
- HomePage – відображає вступну сторінку, глобальний пошук та перелік популярних студій.
- StudiosPage – реалізує сітку карток музичних майданчиків з можливістю фільтрації за містом чи іншими атрибутами пошуку.
- StudioDetailPage – відображає детальні характеристики студії, контактні дані, перелік доступних кімнат та інвентарю.
- BookingPage – форма для обрання слотів у календарі та оформлення резерву на обрану кімнату.
- BookingsListPage – перелік бронювань користувача із табличним виглядом статусів.
- AdminPage – сторінка управління (dashboard) адміністратора студії для модифікації ресурсів, читання розкладу та генерації фінансових звітів.

Стилізовані елементи (Tailwind CSS) адаптуються до розміру екрану. Навігаційне меню (Navbar) та нижня частина сайту ізольовано в окремий компонент Layout, щоб інкапсулювати структуру коду. Взаємодія з сервером супроводжується зрозумілими повідомленнями для користувача під час завантаження інформації. Захист рутів інтегровано із компонентом RequireRole, що відхиляє спроби неавторизованих користувачів перейти до панелі адміністратора.
