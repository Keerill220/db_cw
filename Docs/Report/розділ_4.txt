РОЗДІЛ 4. ОПРАЦЮВАННЯ ДАНИХ

4.1. Характеристика первинного джерела
Первинним джерелом даних для початкового наповнення каталогів та контенту інформаційної системи є вбудований програмний модуль генерації даних (DataSeeder.cs). Замість використання зовнішніх публічних API, проєкт реалізує самодостатній механізм автоматичного наповнення бази даних, що дозволяє миттєво розгорнути повноцінне демонстраційне середовище з репрезентативним обсягом сутностей.

Дані, які автоматично генеруються та завантажуються системою, охоплюють наступні ключові аспекти:
Довідкова інформація: статичні списки міст із зазначенням країн (наприклад, "Lviv, Ukraine", "Warsaw, Poland") та базових категорій музичного обладнання (наприклад, "Drums", "Guitar Amp", "Microphone").
Суб'єкти платформи: автоматична генерація облікових записів клієнтів (близько 800 профілів) та адміністраторів студій, включаючи створення "Superadmin" та тестових власників з ініціалізованими паролями.
Студії та розклад: алгоритмічно згенеровані 200 профілів студій зі специфічними назвами (наприклад, "Sound Wave", "Beat Factory"), адресами та описовим текстом. Для кожної студії створюються 1-5 кімнат із зазначенням площі та вартості години оренди.
Обладнання: випадковий розподіл музичного функціоналу між кімнатами та студіями для можливості оренди.
Фіксація бронювань: масштабне заповнення таблиці бронювань (до 5000 записів), що імітують минулу та майбутню активність клієнтів.

Цей підхід забезпечує високу якість тестування звітів та пошукових фільтрів одразу після компіляції серверного застосунку.

4.2. Алгоритм трансформації даних при завантаженні
Процес ініціалізації та трансформації даних відбувається на етапі запуску сервера (за умови порожньої бази даних) і включає наступні кроки:

Підготовка та перевірка:
Перед початком процесу серверна частина звертається до контексту `ApplicationDbContext` для перевірки наявності існуючих записів у таблиці адміністраторів. Якщо база даних вже містить записи, генерація пропускається, що запобігає дублюванню.

Хешування паролів (Трансформація безпекових даних):
Щоб запобігти надмірному навантаженню на процесор під час масового створення користувачів (генерація понад 850 хешів bcrypt), алгоритм одноразово обчислює криптографічні хеші для трьох типових паролів ("Admin123!", "Owner123!", "Client123!") через сервіс `IPasswordHasher` та зберігає їх у змінні. Далі ці хеші застосовуються до всіх згенерованих об'єктів користувачів.

Генерація та мапінг структур (Нормалізація):
На основі масивів з іменами, прізвищами та назвами студій працює генератор псевдовипадкових чисел `Random`. 
- Міста та типи обладнання зберігаються першими для формування зовнішніх ключів.
- Сутності `Administrator` об'єднуються зі згенерованими об'єктами `Studio`. Зв'язок відбувається через встановлення властивостей `AdminId` та `CityId`.
- Для кожної кімнати додається рандомізоване обладнання, де ціна оренди може динамічно змінюватись.

Транзакційне збереження:
Алгоритм використовує метод поступового накопичення об'єктів у Entity Framework Core (`_db.Studios.AddRange(studios)`).
Після кожного завершеного логічного блоку даних виконується виклик `await _db.SaveChangesAsync()`, що ініціює SQL-транзакцію. Даний підхід гарантує цілісність посилань та дозволяє наступному блоку (наприклад, кімнатам) використовувати вже згенеровані первинні ключі (наприклад, ID студій). Під час останнього кроку – створення бронювань – автоматично спрацьовують тригери бази даних, включаючи розрахунки вартості (`fn_booking_calc_total_price`).
