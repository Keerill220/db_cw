РОЗДІЛ 2. РОЗРОБКА МОДЕЛЕЙ ЗБЕРІГАННЯ ДАНИХ

2.1. Концептуальне проектування

Концептуальна модель інформаційної системи «SoundSpace» включає сукупність сутностей, що відображають ключові об'єкти предметної області, їх властивості та взаємозв’язки. Головною метою цієї моделі є формалізація бізнес-правил та визначення структури бази даних для ефективного управління ресурсами репетиційних баз та студій звукозапису.

Основні сутності та поняття даної предметної області:
Студія (Studio) – центральна сутність, що репрезентує музичну базу або студію та містить інформацію про її розташування.
Місто (City) – довідкова сутність для географічної прив'язки студій.
Кімната (Room) – сутність, що описує окреме приміщення всередині студії із зазначенням площі та вартості оренди.
Обладнання (Equipment) – сутність для обліку музичних інструментів та апаратури, що належать студії. Може мати прив'язку до конкретної кімнати або бути доступним для оренди окремо.
Тип обладнання (EquipmentType) – довідкова сутність для класифікації інвентарю музичної бази.
Адміністратор (Administrator) – сутність, що репрезентує суб'єкта, який управляє профілем студії та її ресурсами.
Клієнт (Client) – сутність, що репрезентує суб'єкта, який бронює музичні кімнати.
Бронювання (Booking) – сутність, що фіксує факт резервування конкретної кімнати визначеним клієнтом на відповідний час та дату, із розрахованою загальною вартістю.
Бронювання обладнання (BookingEquipment) – асоціативна сутність для зв'язку між бронюванням та додатково замовленим обладнанням.

Взаємозв’язки між сутностями визначають структурну логіку системи. Зв'язок "один-до-багатьох" між Студією та Кімнатами, а також Студією та Обладнанням формує каталог послуг адміністратора. Відношення "один-до-багатьох" між Клієнтом та Бронюванням, а також між Кімнатою та Бронюванням формує основний бізнес-процес оренди. Асоціативна таблиця Бронювання обладнання реалізує зв'язок "багато-до-багатьох" між Бронюванням та Обладнанням. Концептуальна діаграма моделі даних відображена на «Рис. 2.1.».

2.2. Вимоги до системи накопичення даних

2.2.1. Оцінка обсягів даних, частота додавання
Інформаційна система працює з даними, що відрізняються частотою модифікації. Довідкові сутності (міста, типи обладнання) та дані, що описують конфігурацію студій (інформація про самі студії, їх кімнати, наявне обладнання), оновлюються відносно рідко і переважно адміністраторами під час налаштування профілю або зміни тарифів. Основний обсяг даних генерується користувачами під час бронювання приміщень. Відповідно, збереження записів у таблиці бронювань та асоціативній таблиці бронювання обладнання відбуватиметься найчастіше. Архітектура бази даних розроблена з урахуванням поступового розширення обсягів даних під час щоденної експлуатації та передбачає оптимізацію доступу для частих операцій читання розкладів.

2.2.2. Очікувані запити та вибірки
Основні сценарії взаємодії із системою вимагають виконання різноманітних запитів до бази даних.

Для клієнтів: Найчастішими є запити на отримання списків доступних студій та приміщень із фільтрацією за містом, ціною або датою. Крім того, регулярно виконуються вибірки розкладу конкретної кімнати для відображення зайнятих та вільних годин, а також отримання історії власних бронювань.

Для адміністраторів: Часто виконуються запити для отримання переліку бронювань їхніх студій, а також вибірки, необхідні для генерації звітів про завантаженість приміщень та статистику доходів за обраний період.

Для прискорення цих вибірок використовуються індекси для полів, які найчастіше зустрічаються в умовних виразах пошуку та об'єднаннях (наприклад, статуси бронювань, ідентифікатори кімнат та дати бронювань).

2.2.3. Обмеження доступу
Захист конфіденційної інформації (персональні дані клієнтів, фінансова статистика студій) є критично важливим аспектом. У межах проєкту розмежування прав доступу реалізоване на основі ролей користувачів: "Admin" та "Client".

Обмеження доступу імплементовано на рівні веб-API:
Автентифікація шляхом застосування JSON Web Tokens (JWT): Кожен користувач після успішної перевірки облікових даних отримує зашифрований токен, який додається до заголовків HTTP-запитів.
Рольова авторизація: Перевірка прав доступу до конкретних маршрутів API реалізується через атрибути авторизації (наприклад, [Authorize(Roles = Roles.Admin)]). Так, операції з додавання нових кімнат чи керування обладнанням доступні лише адміністратору конкретної студії, тоді як створення бронювань вимагає авторизації, але без вимоги специфічної ролі адміністратора. Бізнес-логіка додатково гарантує, що адміністратор отримує доступ лише до своїх студій, а клієнт – лише до своїх бронювань.

2.3. Логічне проектування схеми бази даних
На цьому етапі концептуальна модель перетворена у логічну схему реляційної бази даних "SoundSpace", адаптовану для роботи під управлінням СУБД PostgreSQL. Структура БД приведена до третьої нормальної форми (3НФ), що унеможливлює надмірність дублювання даних та аномалії оновлення.

Спроектовані таблиці:
cities: Довідник міст (id первинний ключ, name).
equipment_types: Довідник категорій обладнання (type_id первинний ключ, name).
clients: Таблиця користувачів з роллю клієнта (client_id первинний ключ, first_name, last_name, email, password_hash, phone).
administrators: Таблиця користувачів з роллю адміністратора (admin_id первинний ключ, first_name, last_name, email, password_hash, is_super_admin).
studios: Таблиця реєстрації музичних баз (studio_id первинний ключ, admin_id зовнішній ключ, city_id зовнішній ключ, name, address, description, is_active).
rooms: Таблиця приміщень (room_id первинний ключ, studio_id зовнішній ключ, name, description, area, price_per_hour, is_active).
equipment: Таблиця музичного інвентарю (equipment_id первинний ключ, studio_id зовнішній ключ, type_id зовнішній ключ, name, price_per_hour, room_id зовнішній ключ). Може бути прив'язане до кімнати або студії загалом.
bookings: Таблиця фіксації оренди (booking_id первинний ключ, room_id зовнішній ключ, client_id зовнішній ключ, date, start_time, end_time, total_price, status (перерахування Pending, Confirmed, Cancelled, Completed)).
booking_equipment: Проміжна таблиця, що пов'язує бронювання з орендованим інвентарем (booking_id первинний ключ та зовнішній ключ, equipment_id первинний ключ та зовнішній ключ, price_at_booking, quantity).

2.4. Реалізація процедур бізнес-логіки
Для забезпечення цілісності та узгодженості даних на рівні СУБД реалізовано декілька тригерів та збережених функцій мовою PL/pgSQL:
fn_booking_no_overlap: Тригерна функція, що активується перед вставкою або оновленням бронювання. Вона перевіряє таблицю на наявність перетинів часових інтервалів для конкретної кімнати та генерує виняток, якщо виявлено накладання.
fn_booking_calc_total_price: Тригер, що автоматично перераховує поле total_price у бронюванні, враховуючи тривалість оренди кімнати (за її тарифом) та вартість доданого обладнання.
fn_equipment_price_snapshot: Тригер, який при додаванні обладнання до бронювання фіксує його поточну ціну в полі price_at_booking.
proc_occupancy_report: Збережена процедура для формування аналітичного звіту завантаженості кімнат студії (загальна кількість годин бронювання та сумарний дохід) за вказаний період.

2.5. Використані інші моделі даних
Взаємодія програми з базою даних побудована з використанням ORM інфраструктури Entity Framework Core. Управління змінами у схемі бази даних здійснюється за допомогою системи EF Core Migrations (програмний підхід Code-First). 
Таблиця __EFMigrationsHistory зберігає лог виконаних міграцій, забезпечуючи точну синхронізацію між C# класами (сутностями) та фізичними таблицями в PostgreSQL. Початкова ініціалізація бази даних та наповнення тестовими даними здійснюються комбінацією автоматично згенерованих міграцій та сирих SQL-скриптів (у міграції ApplySqlScripts), що інкапсулюють створення тригерів та збережених процедур.
